# github licence

| **特性对比**     | **MIT License**                   | **Apache License 2.0**          | **GPL (e.g., v3)**                          |
| ---------------- | --------------------------------- | ------------------------------- | ------------------------------------------- |
| **核心理念**     | **极致宽松** 最大程度推广代码使用 | **商业友好** 提供明确的专利保护 | **强开源（Copyleft）** 确保衍生作品持续开源 |
| **允许闭源商用** | ✅ 是                              | ✅ 是                            | ❌ 否                                        |
| **专利授权**     | ❌ 无明确规定                      | ✅ 有，且提起专利诉讼则授权终止  | ⚠️ GPLv3有相关条款                           |
| **“传染性”**     | 无                                | 无                              | **强**，衍生作品必须开源                    |
| **主要义务**     | 保留版权声明                      | 保留版权声明、标注修改          | 衍生作品必须开源、保留协议和版权            |
| **典型项目**     | Vue.js, React, Node.js            | Kubernetes, Android             | Linux Kernel, GIMP                          |

### 💡 如何添加许可证

添加许可证的操作非常简单，主要有两种方式：

- 

  **创建新仓库时添加**：在GitHub上创建新仓库的页面，填写基本信息后，你会看到一个 **“Add a license”** 的下拉菜单。直接从列表中选择合适的许可证（如MIT License），这样仓库创建后就会自动包含LICENSE文件。这是最推荐的方式。

- 

  **为已有仓库添加**：进入你的仓库，点击“Add file” -> “Create new file”，在文件名输入框中键入 **`LICENSE`**（全大写）。输入后，页面右侧会出现一个“Choose a license template”按钮，点击它即可从模板中选择并自动生成。

### 💎 总结与建议

简单来说，**为项目添加许可证是为其颁发一张“使用说明书”和法律保障**。没有许可证的项目默认是“保留所有权利”，这会阻碍他人合法使用和贡献。

对于大多数开发者，特别是个人项目或希望代码被广泛使用的场景，**MIT许可证通常是最直接、最受欢迎的选择**。如果你的项目有更特定的需求，可以参考上方的流程图和表格进行比较。
